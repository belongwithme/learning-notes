Git 与预览发布工作流

把“直接改线上”升级成“先彩排、再正式演出”的安全习惯。

Preview 是彩排，Production 是正式演出

Preview Deployment 不会替换正式域名；Production Deployment 才会更新 notes.ian2000.xyz。main 应被当作随时可以发布的稳定主线。

🎭

功能分支 / PR

每次推送生成独立预览 URL。你可以先检查页面、搜索和移动端样式。

🎬

main 分支

推送或合并后触发生产部署，READY 后自定义域名切换到新版本。

🛡️
推荐原则：main 永远保持可上线

即使是个人项目，也优先在分支拿到预览地址，验证后再合并。这样失败只影响彩排，不影响正在访问网站的人。

一次稳妥改版的六步节奏

分支 像独立排练厅；PR 是正式演出前的验收单。

1
从 main 创建功能分支

例如 codex/improve-search，改动先与生产版本隔离。

2
本地检查与构建

运行 npm run check 和 npm run build，提前拦截基础错误。

3
提交并推送分支

GitHub 收到新快照，Vercel 自动创建 Preview Deployment。

4
打开预览 URL 验证

检查核心路径、手机尺寸、交互和控制台报错。

5
创建 PR 并等待检查

确认 GitHub Actions 与 Vercel 两边都通过。

6
合并 main

Vercel 生成生产部署，域名切换后再做一次线上抽查。

你的仓库已经有一道自动质量门

CI 工作流同时监听 PR 和 main。也就是说：预览前能检查，正式部署前还能再检查一次。

.github/workflows/validate.yml 
on:
  pull_request:
  push:
    branches:
      - main

jobs:
  validate:
    runs-on: ubuntu-latest

Git 自动部署是主路，Vercel CLI 是工具箱

Vercel CLI 不必替代 Git 工作流。日常用 Git 推送；需要查询、临时预览、拉取环境变量或回滚时，再打开命令行工具箱。

npm i -g vercel安装 Vercel 命令行工具。
vercel login通过浏览器完成账号登录。
vercel link --yes --project learning-notes --scope ian2000明确把当前目录连接到你的项目与团队。
vercel从当前代码创建一个预览部署。
vercel ls列出最近的部署及状态。
vercel inspect <url>检查某次部署的元数据与构建信息。
vercel --prod直接创建生产部署；日常更推荐通过 main 分支触发。
vercel rollback把生产域名退回上一份可用成品。
⚠️
不要同时发起两条生产路径

如果 GitHub main 正在自动部署，就不要又手动运行 vercel --prod。否则会同时出现两次生产部署，排查“到底哪次生效”会变得困难。
