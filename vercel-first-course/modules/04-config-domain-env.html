配置、域名与环境变量

学会管理站点的“配电间”：运行版本、固定入口和不能写进代码的秘密。

三类东西，放在三个位置

代码仓库像公开施工图，Vercel Settings 像配电间，环境变量 像上锁的柜子。放错位置，轻则构建失败，重则泄露密钥。

⑂

仓库内配置

package.json、astro.config.mjs、工作流；可审查、可追踪，不放秘密。

⚙

Project Settings

框架、根目录、构建命令、Node 版本、Git 连接和访问保护。

🔐

Environment Variables

API 密钥等敏感值，并按 Production、Preview、Development 隔离。

真实隐患：本地用 Node 22，Vercel 项目现在是 24.x

你的仓库声明“至少 22.12.0”，而不是锁定 22 主版本。Node.js 的主版本可能带来不兼容变化；最新构建日志已经提示 Vercel 将来可能自动升级。

package.json 
{
  "name": "learning-notes",
  "type": "module",
  "version": "0.0.1",
  "private": true,
  "engines": {
    "node": ">=22.12.0"
  },

⚠️
版本漂移不会立刻报错，但会积累不确定性

GitHub Actions 使用 Node 22，Vercel 项目显示 24.x。最好让 package.json、CI 与 Vercel Project Settings 使用同一主版本，再由你主动安排升级。

环境变量必须按环境隔离

作用域 就像三套不同钥匙：预览版本不应该拿到生产数据库的万能钥匙。你的当前站点是纯静态 Astro 站，暂时没有业务密钥，但未来接入 API 时必须先守住这条边界。

Production正式域名使用；只放线上真实配置。
Preview分支和 PR 使用；应连接测试资源或使用受限密钥。
Development本地开发使用；通过 vercel env pull 拉到本机。
PUBLIC_*Astro 会允许客户端代码读取；它会暴露给浏览器，绝不能放秘密。
无 PUBLIC_ 前缀用于服务端或构建过程；仍需避免把值手动写进输出页面。

域名是固定门牌，部署是门后的房间

自定义域名 不保存网页文件。它通过 DNS 把访客送到 Vercel，再由 Vercel 把流量交给当前生产部署；HTTPS 证书则保护传输过程。

1
在 Domains 添加域名

你的项目目前已经绑定 notes.ian2000.xyz。

2
按提示配置 DNS

如果域名不由 Vercel 托管，就在原 DNS 服务商添加 Vercel 要求的记录。

3
等待验证与 HTTPS

Vercel 验证所有权并自动签发证书，状态正常后浏览器显示安全连接。

4
生产发布更新指向

新部署 READY 后，固定门牌指向新的生产成品。

notes.ian2000.xyz当前主自定义域名。
learning-notes-ian2000.vercel.appVercel 提供的稳定项目域名。
learning-notes-1r6gnwvor-ian2000.vercel.app这一次具体部署的不可变地址。
