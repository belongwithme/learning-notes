日志、排障与回滚

出问题时先分诊，再取证，最后决定修复、重部署还是回滚。

像急诊分诊一样，先判断故障发生在哪一段

不要看到“网站不对”就立刻重写代码。先把症状分到构建、运行、域名或性能四个区域，再去拿对应的 日志 和状态证据。

🔨

你的仓库与 Vercel，其实在做两次互补检查

GitHub Actions 先在独立机器上验证项目；Vercel 再在自己的构建环境生成发布成品。两边的日志可以帮助你判断：问题来自代码本身，还是只在云端配置中出现。

出事故时：先止血，再修复历史

Instant Rollback 很像把电闸切回上一套稳定线路。它能迅速恢复访问，但 Git 的 main 仍可能保留有问题的提交。

1
确认影响范围

记录故障页面、开始时间和当前生产 Deployment。

2
回滚到上一份 READY 成品

Dashboard 使用 Instant Rollback，或终端运行 vercel rollback。

3
验证域名恢复

访问核心页面并观察错误率，不要只相信按钮提示。

4
修复 Git 历史中的问题

创建修复提交或安全地撤销坏提交，避免下一次推送把故障重新带回来。

5
重新走 Preview → Production

验证新成品，再恢复正常发布节奏。

🧠
回滚恢复服务，修复恢复事实

只回滚不修 Git，会留下“线上稳定、main 有毒”的分裂状态。下一次自动部署可能再次把问题带回来。

你的 Vercel 日常操作清单

开发前从最新 main 建分支；如果项目用了秘密，再同步 Development 环境变量。
推送前npm run check && npm run build。
Preview 后打开关键路径，检查桌面与手机，确认 GitHub Actions 通过。
Production 后确认提交、READY、自定义域名和核心页面。
出问题先分诊：Build Logs、Runtime Logs、Domains 或 Speed Insights。
影响用户先回滚止血，再修复 main 并重新发布。
